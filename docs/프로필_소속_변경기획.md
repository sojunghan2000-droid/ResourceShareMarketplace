# 가입 소속 선택 + 개인정보 편집(본인/관리자) — 변경 기획

| 항목 | 내용 |
|---|---|
| 작성일 | 2026-06-17 |
| 대상 | Streamlit(`app.py`·`core/`) + React(`web/`) + Supabase(RLS·RPC) |
| 변경 | ① 가입 시 소속(협력사) 선택 ② 본인 프로필 편집 ③ 관리자의 사용자 정보 편집 |

---

## 1. 현황

- 가입: 아이디·이름·비밀번호만. **소속은 관리자가 승인 시 지정**(가입자는 자기 소속을 못 밝힘).
- 프로필: 헤더 사용자 메뉴에 이름·소속·역할 **표시만**, 편집 없음.
- `app_users` 컬럼: `id, org_id, name, phone(미사용), role, status, created_at`.
- ⚠️ **보안 허점 발견**: 현재 RLS `users_self_update` 가 본인 행의 **모든 컬럼** 수정 허용 → 사용자가 자기 `role`을 admin 으로, `status`를 active 로 바꿀 수 있음(권한 상승). 이번 작업에서 **반드시 수정**.

## 2. 목표 구체화 — "개인정보"와 편집 권한 매트릭스

"개인정보"의 범위와 **누가 무엇을 바꿀 수 있는지**를 먼저 못 박는다.

| 필드 | 본인(프로필) | 관리자 | 비고 |
|---|---|---|---|
| 이름 | ✅ | ✅ | |
| 연락처(phone) | ✅ | ✅ | 현재 미사용 컬럼 활용 |
| 비밀번호 | ✅ (본인 변경) | ✅ (변경/초기화) | 본인=auth.updateUser, 관리자=admin_set_password(기구현) |
| 소속(협력사) | ❌ (또는 '변경요청') | ✅ | **결정 §8-2** |
| 역할(role) | ❌ | ✅ | member/admin |
| 상태(status) | ❌ | ✅ | pending/active/disabled |
| 아이디(ID/email) | ❌ | ❌ | 불변(로그인 식별자) |

> 핵심 원칙: **본인은 이름·연락처·비밀번호만**. 소속·역할·상태는 **관리자 전용**(권한·조직 무결성).

## 3. 기능별 상세

### 3.1 가입 시 소속(협력사) 선택
- 가입 폼: 아이디·이름·**소속(드롭다운)**·비밀번호.
- 선택한 협력사를 `app_users.org_id` 에 **요청값으로 저장**(status=pending 유지).
- **관리자 승인 화면**: 가입자가 선택한 소속을 **기본값으로 표시** → 관리자가 확인/수정 후 승인.
- ⚠️ **익명 사용자도 협력사 목록을 읽어야 함** — 현재 `organizations` RLS는 `to authenticated`라 anon이 못 읽음.
  → **공개 RPC `list_orgs_public()`**(SECURITY DEFINER, id+name 만 반환) 추가로 노출 최소화. RLS는 그대로 유지.
- `register_user` RPC 시그니처에 `p_org_id` 추가(유효성: organizations 존재 확인).

### 3.2 본인 프로필 편집
- 진입: 헤더 **사용자 메뉴 → "내 프로필"** → 모달/페이지.
- 편집: 이름, 연락처, (비밀번호 변경 섹션).
- 저장: **`update_my_profile(p_name, p_phone)` RPC**(SECURITY DEFINER, 본인 행의 name/phone 만 갱신) — 컬럼 화이트리스트로 role/status/org 변조 차단.
- 비밀번호: `supabase.auth.updateUser({password})`(본인 세션). 현재 비번 확인 단계(선택).
- **RLS 수정(필수)**: `users_self_update` 를 제거하거나 name/phone 로 제한. 직접 update 대신 RPC 경유로 통일 권장.

### 3.3 관리자의 사용자 정보 편집
- 위치: 관리자 센터 **사용자 관리**(기존 비번변경/초기화 옆에 "정보 수정").
- 편집: 이름·연락처·소속·역할·상태.
- 저장: 관리자 RLS(`users_admin_all`)로 직접 update 또는 `admin_update_user(...)` RPC.
- (이미 있음) 비밀번호 변경/초기화.

## 4. 백엔드 변경 요약
| 구분 | 변경 |
|---|---|
| RLS(보안) | `users_self_update` 제거/축소 → 자기 role·status·org 변조 차단 |
| RPC | `list_orgs_public()` (anon, 협력사 목록) |
| RPC | `register_user(p_id,p_pw,p_name, p_org_id)` 로 확장(소속 저장) |
| RPC | `update_my_profile(p_name,p_phone)` (본인, definer) |
| RPC(선택) | `admin_update_user(p_user,p_name,p_phone,p_org,p_role,p_status)` 또는 관리자 직접 update |
| 데이터 | `phone` 컬럼 실사용 시작 |

## 5. UI 변경 요약
| 화면 | 변경 |
|---|---|
| 가입 폼(Login) | 소속 드롭다운 추가(공개 org 목록) |
| 헤더 사용자 메뉴 | "내 프로필" 항목 → 편집 모달(이름·연락처·비번) |
| 관리자 승인 대기 | 가입자 선택 소속을 기본값으로 |
| 관리자 사용자 관리 | 각 사용자 "정보 수정"(이름·연락처·소속·역할·상태) |

## 6. 보안 주의 (필수 반영)
1. **자기 권한 상승 차단** — self-update를 name/phone 로만 제한(RPC 화이트리스트). 현 RLS 허점 제거.
2. **소속/역할/상태는 관리자만** — 본인 경로에서 절대 변경 불가.
3. **공개 org 목록**은 id+name 만(다른 컬럼 노출 금지).
4. 가입 시 선택 소속은 **요청값** — 관리자 승인으로 최종 확정(자가 신고 신뢰 안 함).

## 7. 단계 / 공수(예상)
| 단계 | 내용 | 공수 |
|---|---|---|
| P1 | RLS 허점 수정 + `update_my_profile`·`list_orgs_public` RPC | 0.5d |
| P2 | 가입 폼 소속 선택 + `register_user` 확장 + 승인화면 기본값 | 0.5d |
| P3 | 헤더 "내 프로필" 편집 모달(이름·연락처·비번) — 양 앱 | 0.5d |
| P4 | 관리자 사용자 정보 수정 UI — 양 앱 | 0.5d |

> 총 ~2일. **P1(보안 수정)은 다른 작업과 무관하게 우선 적용 권장.**

## 8. 확인 필요 (Open)
1. **개인정보 필드 범위** — 이름·연락처 외 추가(이메일/부서/직책 등)? *(권장: 우선 이름·연락처만)*
2. **본인 소속 변경** — 아예 불가(관리자만) vs "변경 요청"(승인 큐)? *(권장: 우선 관리자만, 요청 기능은 후순위)*
3. **가입 소속 선택 필수 여부** — 필수 vs 선택(미선택 시 관리자가 지정)?
4. **비밀번호 변경 시 현재 비번 확인** 단계 둘지.
5. **관리자 정보수정 UI 형태** — 인라인 vs 모달(팝오버).

### 권장 결론
- **P1(보안 RLS 허점 수정)** 을 최우선 — 현재 누구나 자기 역할을 admin 으로 올릴 수 있음.
- 편집 권한은 매트릭스(§2)대로: **본인=이름·연락처·비번 / 관리자=전체**.
- 가입 소속은 **자가 선택 + 관리자 확정** 2단계로, 익명 org 목록은 전용 공개 RPC로 안전하게 노출.
- 결정 §8 5개(특히 1·2)만 정해지면 P1~P4 착수.
