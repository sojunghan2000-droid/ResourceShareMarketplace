| 항목 | 내용 |
|---|---|
| 작성일 | 2026-06-17 |
| 성격 | **실제 구현 기준 단일 출처(SSOT)**. PRD §8 요약표를 대체·확장 |

| 범위 | 역할·주체·권한 매트릭스·테이블별 RLS·RPC 권한·인증·알려진 이슈 |

## 1. 주체(Subject)와 역할(Role)

| 구분 | 값 | 설명 |
|---|---|---|
| 역할 `role` | `member` / `admin` | member=협력사 사용자(대여자·신청자 겸), admin=원청 운영자 |
| 상태 `status` | `pending` / `active` / `disabled` | active 만 앱 사용. pending=승인대기 |
| 소속 `org_id` | 협력사(organizations) | 데이터 격리 단위. admin 은 전체 |

- **인증 식별자**: ID(=내부 이메일 `id@safeshare.app`). 불변.
- member 는 **자재 단위로** 대여자(보유측)·신청자(차용측)가 동시에 될 수 있음.
- admin 은 **조직 경계를 넘어 전체** 조회·처리.

## 2. 권한 매트릭스 (동작 × 역할) — 현행 기준

| 동작 | member(보유측) | member(신청측) | admin |
|---|---|---|---|
| 자재 검색/목록 | ✅ | ✅ | ✅ |
| 자재 등록·수정·삭제 | ✅ **본인 조직** | — | ✅ 전체 |
| 대여 신청 | — | ✅ (자기 조직 자재 제외) | ✅ |
| 승인 / 거절 | ✅ **본인 자재** | — | ✅ **전체** |
| 수령 확인(사진·서명) | — | ✅ (신청자) | ✅ |
| 반납 최종 확정 | ✅ **보유자** | — | ✅ **전체** |
| 대여 건 조회 | ✅ 본인 조직 관련 | ✅ 본인 신청 | ✅ **전체** |
| 대시보드 | 본인 조직 범위 | 본인 조직 범위 | **전체** |
| 알림 조회/읽음 | ✅ 본인 | ✅ 본인 | ✅ 본인 |
| **가입 승인** | — | — | ✅ |
| **사용자 비번 변경/초기화** | — | — | ✅ |
| **표준단가 편집** | — | — | ✅ |
| **연체 독촉 발송** | — | — | ✅ |
| 본인 비밀번호 변경 | ✅ | ✅ | ✅ |
| 본인 프로필(이름·연락처) | (예정) | (예정) | (예정) |

> 표시값은 **실제 RLS+RPC로 집행**됨(아래 §3·§4). member 의 "전체" 동작은 없음 — 항상 조직/본인 범위.

## 3. 테이블별 RLS 정책 (행 접근)

모든 정책은 `to authenticated`(익명 차단). admin 판별 = `is_admin()`, 소속 = `current_org_id()`.

| 테이블 | SELECT | INSERT/UPDATE/DELETE |
|---|---|---|
| `organizations` | 로그인 사용자 전체 | admin 만(`org_admin_all`) |
| `app_users` | 본인 또는 admin | **본인 일부**(⚠️§6) / admin 전체 |
| `materials` | active 사용자 전체(검색용) | **본인 조직** 또는 admin |
| `loans` | lender_org 또는 borrower_org 소속, admin 전체 | **직접 불가** → RPC 전용 |
| `loan_events` | 해당 대여 당사자 또는 admin | 직접 불가(RPC) |
| `notifications` | 본인 | 본인(읽음 처리) |
| `categories` | 전체(true) | (seed 고정) |
| `category_price` | 로그인 전체 | admin 만(`cp_admin`) |
| Storage(`material-photos`,`loan-proofs`) | authenticated | authenticated |

- **익명(anon)**: `categories` 와 가입용 공개 목록 외 접근 불가.
- **타 조직 데이터 노출 0** (loans/materials 격리).

## 4. RPC 권한 (상태전이·권한작업) — SECURITY DEFINER

상태전이는 RLS 직접 변경이 아니라 **RPC로만**(원자성·경합방지·권한검사 내장).

| RPC | 호출 가능 | 내부 권한 검사 |
|---|---|---|
| `request_loan` | authenticated | 신청자=본인, 자기조직 자재 금지, 안전상태 차단 |
| `approve_loan`/`reject_loan` | authenticated | lender_org = 본인 조직 **또는 admin** |
| `pickup_loan` | authenticated | borrower = 본인 **또는 admin** |
| `request_return` | authenticated | borrower = 본인 **또는 admin** |
| `return_loan` | authenticated | lender_org = 본인 조직 **또는 admin** |
| `mark_overdue_loans` | authenticated | (멱등 유지보수) |
| `send_overdue_reminders` | authenticated | **`is_admin()` 필수** |
| `admin_set_password` | authenticated | **`is_admin()` 필수** |
| `register_user` | **anon** + authenticated | 형식·중복 검증(가입 엔드포인트) |
| `current_org_id`/`is_admin`/`is_active` | authenticated | RLS 평가용 헬퍼 |
| `_notify`/`_notify_lender` | **차단**(내부 전용) | definer 내부 호출만 |

- 모든 definer 함수 `search_path` 고정(하이재킹 방지). pgcrypto 사용 함수는 `extensions` 포함.

## 5. 인증·계정 권한

- **로그인/가입**: ID/PW. 가입 즉시 `pending` → **admin 승인** 시 `active` + 소속 확정.
- **비밀번호**: 본인=`auth.updateUser`(예정 UI) / admin=`admin_set_password`(변경·초기화 `1111`).
- **역할 부여(admin 승격)**: 현재 **SQL/관리자 전용**. UI 미노출(의도).

## 6. ⚠️ 알려진 권한 이슈 (수정 대상)

| 이슈 | 내용 | 영향 | 조치 |
|---|---|---|---|
| **자기 권한 상승** | `users_self_update` 가 본인 행 **전체 컬럼** 허용 → 사용자가 자기 `role`을 admin, `status`를 active 로 변경 가능 | **권한 상승(심각)** | self-update를 name/phone 로만 제한(RPC 화이트리스트) — [프로필_소속_변경기획](프로필_소속_변경기획.md) P1 |
| 남은 advisor 경고(0029) | authenticated 가 definer RPC 호출 가능 | 정보성(의도된 설계) | 유지 |

> **이 §6-1은 가능한 빨리 적용 권장.**

## 7. 설계 원칙

1. **이중 방어**: 앱 레벨 체크 + DB RLS/RPC. 클라이언트 우회 불가.
2. **조직 격리**: member 는 항상 본인 조직/본인 데이터. 교차 접근은 admin 만.
3. **상태전이는 RPC 단일 경로**: 직접 UPDATE 금지 → 무결성·이력·알림 보장.
4. **최소 노출**: 익명은 categories/가입목록만. 내부 헬퍼는 호출 차단.
5. **불변 식별자**: ID(이메일)·loan_events(이력)는 변경 불가.

### 요약
- 권한은 **역할(member/admin) + 소속(org) + 상태(active)** 3축으로, **RLS+RPC가 실제 집행**.
- member=조직·본인 범위, admin=전체. 상태전이·권한작업은 RPC 권한검사 내장.
- **단 하나의 심각 이슈(자기 역할 변경)는 즉시 수정 필요** — 본 문서가 그 추적·근거.
