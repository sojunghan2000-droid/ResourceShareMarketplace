## 1. 배경 (Background)

플랜트/제조 현장에서는 공사·정비 물량에 따라 **안전자재(가설 안전가시설·보호구류)** 의 수요가 협력사·공구·시기별로 출렁인다. 그 결과:

- A협력사는 공정이 끝나 **잉여 안전난간·단관비계**가 창고에 묵혀 있고,
- B협력사는 다음 주 작업에 같은 자재가 **부족**해 신규 구매·임차를 한다.

현재 이 수급 불일치는 **단체 카톡방**에서 "○○ 남는 분 계신가요?" 식의 텍스트로 해소되고 있다. 이 방식의 근본 문제:

1. **재고 가시성 없음** — 누가 무엇을 얼마나 갖고 있는지 한눈에 안 보임. 스크롤하다 묻힘.
2. **이력 추적 불가** — 누가 빌려갔는지, 언제 돌려줘야 하는지 기록이 흩어짐. 분실·미반납 책임 불명확.
3. **검색 불가** — 과거 글을 다시 찾을 수 없음. 같은 질문 반복.
4. **정산/집계 불가** — 공유로 절감한 비용, 자재 회전율 등 성과 측정 불가.
5. **안전자재 특수성 누락** — 검교정일·사용기한·점검상태 같은 **안전 관련 속성**을 텍스트로는 관리 못 함.

## 2. 목표 (Goals)

| # | 목표 | 성공 지표(KPI) |
|---|---|---|
| G1 | 카톡방을 대체하는 **구조화된 잉여자재 게시·검색** 제공 | 월 등록 자재 건수, 검색 사용률 |
| G2 | **신청 → 승인 → 수령 → 반납** 전 과정을 시스템에서 추적 | 대여 건 중 시스템 처리 비율 ≥ 80% |
| G3 | **미반납·연체 가시화**로 자재 회수율 개선 | 정상 반납률, 평균 연체일 |
| G4 | 공유를 통한 **구매·임차 비용 절감액** 정량화 | 분기별 추정 절감액 리포트 |
| G5 | 안전자재 **점검상태/사용기한 관리**로 부적합 자재 대여 차단 | 부적합 대여 0건 |

### Non-Goals (이번 범위 아님)
- 유상 거래·정산·세금계산서 발행 (무상 대여 모델 고정)
- 소유권 이전(양도) — 반납 전제이므로 제외
- 자재 운송/물류 대행 (수령 장소·시간만 협의, 실제 운반은 당사자)
- 외부 일반 대중 공개 (사내/원청 단위 폐쇄망)
- **자유 그룹 채팅(카톡방형 실시간 대화방) — 의도적으로 만들지 않음 (아래 §2.1 근거 참조)**

### 2.1 채팅을 만들지 않는 이유 (설계 원칙)

본 플랫폼은 카톡방을 **앱 안에 그대로 옮기는 것이 아니라**, 카톡방이 하던 일을 **구조화된 프로세스로 대체**하는 것이 목표다. 카톡방의 본질적 문제는 "대화"가 아니라 **중요 정보가 잡담에 묻혀 검색·추적이 안 되는 것**이었다.

| 카톡방에서 하던 것 | 본 앱의 대체 방식 |
|---|---|
| "○○ 남는 분?" 글 | 자재 등록 (검색 가능, 안 묻힘) |
| "저 빌릴게요" 댓글 | 신청 버튼 (수량·기한 구조화 기록) |
| "언제 가져가요?" 잡담 | 신청 폼의 수령일/용도 필드 |
| "돌려받았어요" 확인 | 반납 확정 + 사진·서명 |
| 누가 뭐 빌렸는지 기억 | 이력·대시보드 자동 추적 |

→ 카톡방 기능의 대부분은 **채팅이 아니라 구조화된 버튼/필드로 대체**된다. **자유 그룹 채팅을 추가하면 탈출하려던 카톡방을 앱 안에 다시 짓는 셈**이 되어, 정보가 다시 묻히고 추적성이 무너진다. 따라서 자유 채팅방은 명시적 Non-Goal로 둔다.

**그렇다면 당사자 간 짧은 문답(규격 확인, 수령 시간 조율 등)은?**
- 0순위: **알림**이 상태 변화를 자동 통지하므로 대부분의 "확인" 대화 자체가 불필요.
- 1순위: **신청 폼의 용도/메모 필드**로 흡수.
- 잔여分(P3 선택): **대여 건별 댓글 스레드**(자유 채팅방 ❌)로 해결 — 해당 자재/대여 건에만 붙고, 맥락과 함께 영구 보관되며, 알림과 연동. §10 릴리스 계획 P3 참조.

## 3. 사용자 & 페르소나 (Personas)

| 역할 | 설명 | 핵심 니즈 |
|---|---|---|
| **자재 보유자 (Lender)** | 협력사 자재담당. 잉여 자재를 등록·대여 승인·반납 확인 | 빠른 등록, 누가 빌려갔는지 추적, 안전한 회수 |
| **자재 신청자 (Borrower)** | 협력사 현장담당. 필요 자재 검색·신청·수령·반납 | 가용 재고 검색, 간편 신청, 반납 알림 |
| **관리자 (Admin / 원청 안전팀)** | 플랫폼 운영, 협력사·사용자 승인, 전체 현황·성과 모니터링 | 통제, 미반납 관리, 성과 리포트, 안전기준 강제 |

> 1인이 Lender이자 Borrower일 수 있음(역할은 자재 단위로 결정).

## 4. 핵심 사용자 시나리오 (User Flow)

### 4.1 잉여자재 등록 (Lender)
1. 로그인 → "내 자재 등록"
2. 품목/규격/수량/사진/보관 위치/점검상태/대여 가능 기간 입력
3. 등록 시 상태 = `가용(AVAILABLE)` → 검색 목록에 노출

### 4.2 신청–수령–반납 프로세스 (핵심)

```
[가용] --신청--> [신청대기] --Lender 승인--> [대여승인]
   |                  |  (거절)                    |
   |                  +--> [가용]으로 복귀          | 수령확인
   |                                               v
   |                                          [대여중] --반납요청--> [반납확인대기]
   |                                               |                      |
   +<-------- Lender 반납완료 처리 ----------------+<---------------------+
                                              [가용]으로 복귀(+이력기록)
```

| 단계 | 행위자 | 동작 | 상태 전이 |
|---|---|---|---|
| ① 신청 | Borrower | 수량·희망 수령일·반납예정일·용도 입력 후 신청 | AVAILABLE → REQUESTED |
| ② 승인/거절 | Lender | 신청 검토 후 승인 또는 거절(사유) | REQUESTED → APPROVED / AVAILABLE |
| ③ 수령확인 | Borrower(신청측) | 실물 인수 시 수령 확인 + **사진·서명 캡처** | APPROVED → ON_LOAN |
| ④ 반납요청 | Borrower | 반납 준비 완료 통보 | ON_LOAN → RETURN_PENDING |
| ⑤ 반납완료 | **Lender(보유측)** | 실물·수량·상태 확인 후 반납 **최종 확정** + **사진·서명 캡처** | RETURN_PENDING → AVAILABLE |

- 각 전이마다 **알림** 발송, **이력(audit log)** 기록.
- **반납예정일 초과** 시 자동 `연체(OVERDUE)` 플래그 + 알림.

#### 반납 확인 정책 (확정)
- **2단계 양측 확인**: 신청자가 "반납 요청"(④)을 올리고, **실물을 받은 보유자가 "반납 완료"로 최종 확정**(⑤). 수량 복원·자재 재가용화는 **⑤ 보유자 확정 시점**에만 발생 → "줬다/못 받았다" 분쟁 차단.
- **증빙 필수**: 수령(③)·반납확정(⑤) 모두 **반납 실물 사진 + 전자서명 캡처**를 첨부해야 처리 완료.
- **반납 수량 조정**: 보유자가 실제 회수 수량을 입력. 빌려준 수량보다 적으면 차액은 `미반납/분실` 플래그로 분리 기록.
- **상태 갱신**: 반납 시 보유자가 점검상태(양호/점검필요/파손)를 지정 → 자재 `inspection_status` 자동 반영.

### 4.3 관리자 모니터링 (Admin)
- 협력사/사용자 가입 승인
- 전체 자재·대여 현황 대시보드(가용/대여중/연체/반납률)
- 연체·미반납 자재 목록 및 독촉
- 성과 리포트(공유 건수, 추정 절감액, 회전율)

## 5. 기능 요구사항 (Functional Requirements)

### 5.1 인증·조직 (Auth & Org)
- FR-A1. 협력사(조직) 단위 회원 구조. 사용자는 1개 협력사에 소속.
- FR-A2. 가입 시 관리자 승인(폐쇄망). 이메일/사번 기반 로그인.
- FR-A3. 역할: `member`(Lender/Borrower 겸용), `admin`.

### 5.2 자재 카탈로그 (Catalog)
- FR-C1. 자재 등록: 카테고리·품목·규격·수량·단위·사진(다중)·보관위치·대여가능기간.
- FR-C2. **안전 속성**: 점검상태(양호/점검필요/사용금지), 검교정·점검일, 사용기한(해당 시).
  - 점검상태 `사용금지` 또는 사용기한 경과 자재는 **신청 불가**(자동 차단).
- FR-C3. 표준 카테고리 사전(아래 §9) 제공 + 기타 직접입력.
- FR-C4. 검색·필터: 카테고리/키워드/협력사/가용여부/위치.
- FR-C5. 수량 부분 대여 지원(예: 보유 50, 신청 20 → 잔여 30 가용).

### 5.3 신청·대여 (Loan Lifecycle)
- FR-L1. §4.2 상태머신대로 신청→승인→수령→반납 처리.
- FR-L2. 신청 시 필수: 수량, 희망 수령일, **반납예정일**, 용도.
- FR-L3. 승인은 Lender 권한. 거절 시 사유 필수.
- FR-L4. 수령(③)은 신청자, **반납 최종 확정(⑤)은 보유자** 권한. 양 단계 모두 **사진 + 전자서명 캡처 필수**. 반납 확정 시 실제 회수 수량 입력(부분 반납 시 차액 = 미반납 플래그) 및 점검상태 지정.
- FR-L5. 반납예정일 경과 시 OVERDUE 처리 및 일일 알림.
- FR-L6. 대여 건별 이력(누가·언제·무엇을·수량) 불변 기록.

### 5.4 알림 (Notifications)
- FR-N1. 상태 전이 시 관련자에게 인앱 알림 + (선택) 이메일.
- FR-N2. 반납 D-3/D-1/D-day, 연체 발생 시 알림.
- FR-N3. (확장) 카카오 알림톡/이메일 다이제스트.

### 5.5 대시보드·리포트 (Dashboard)
- FR-D1. 개인: 내 등록 자재, 내 대여 신청/진행 현황.
- FR-D2. 관리자: 전체 KPI(가용/대여중/연체), 협력사별 현황, 절감액 추정.
- FR-D3. 추정 절감액 = Σ(대여 건의 자재 기준단가 × 수량 × 대여기간 가중). 기준단가는 카테고리별 기본값 테이블.
- FR-D4. CSV/엑셀 내보내기.

## 6. 비기능 요구사항 (Non-Functional)
- NFR-1. **보안/격리**: Supabase RLS로 협력사·역할 단위 행 접근 제어. 타 조직 데이터 노출 금지.
- NFR-2. **모바일 우선**: 현장 사용 다수 → Streamlit 반응형, 큰 터치 타깃, 사진 업로드 최적화.
- NFR-3. **성능**: 목록·검색 1초 내, 캐시 적용(기존 자재관리 앱 패턴 준용).
- NFR-4. **가용성**: Streamlit Cloud 배포, 자동 재기동.
- NFR-5. **감사성**: 모든 대여 전이 audit log 영구 보존.
- NFR-6. **한글 처리**: PDF/리포트 한글 폰트 정상 렌더(NanumGothic 번들).

## 7. 데이터 모델 (개념)

```
organizations (협력사)
  └─ users (소속·역할)

materials (자재)
  - org_id(소유), category, name, spec, qty_total, qty_available,
    unit, location, photos[], inspection_status, inspected_at,
    expires_at, lendable_from, lendable_to, status

loans (대여 건)
  - material_id, borrower_user_id, borrower_org_id, lender_org_id,
    qty, requested_at, pickup_date, due_date, returned_at,
    status(REQUESTED/APPROVED/ON_LOAN/RETURN_PENDING/RETURNED/REJECTED/OVERDUE),
    purpose, reject_reason

loan_events (이력)
  - loan_id, event_type, actor_user_id, payload, created_at

notifications
  - user_id, type, ref_loan_id, message, read_at
```

자세한 스키마/DDL은 개발 기획서 참조.

| 화기·밀폐 | 소화기, 방화포, 가스감지기, 송풍기, 산소농도측정기 |
| 전기안전 | 누전차단기, 접지선, 임시분전반, 절연매트 |
| 기타 | 직접 입력 |

## 10. 릴리스 계획 (Phasing)

| 단계 | 범위 | 목표 |
|---|---|---|
| **MVP (P1)** | 인증·조직, 자재 등록/검색, 신청→승인→수령→반납 상태머신, 기본 알림(인앱), 개인 대시보드 | 카톡방 대체 핵심 |
| **P2** | 관리자 대시보드·KPI, 연체관리, 절감액 추정, CSV 내보내기, 안전속성 차단 로직 | 운영·성과 |
| **P3** | 이메일/알림톡, QR 자재태그, 모바일 PWA 최적화, **대여 건별 댓글 스레드(자유 채팅방 아님, §2.1)** | 편의·확장 |
